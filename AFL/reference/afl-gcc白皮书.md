# 0 设计声明

AFL 尽可能不专注在单个原理操作和特定某个理论证明，这个工具可以认为是hacks工具而且已经在实践中测试，高效，且以最简单的方式实现，当时我能想到最健壮的方式。



得益于为该工具服务的轻量级指令使得很多最终功能变得可能，但是这个机制应该仅仅被认为是达到目的的一种手段。唯一的主要原则，速度，可靠性和易用性。

# 1 覆盖测量

指令注入到编译程序来捕捉覆盖分支 ，以及粗略分支命中技术。在分支点注入的代码在本质等同与

```c
  cur_location = <COMPILE_TIME_RANDOM>;
  shared_mem[cur_location ^ prev_location]++; 
  prev_location = cur_location >> 1;
```



cur_location 是随机生成的来简化链接复杂项目的过程和保持异或输出均匀分布。

shared_mem[] 是一块64kb大小的共享内存区域用来被调用者传递给已装备（应该是指使用afl配套的编译工具编译过）的二进制文件。  

映射的大小是被指定的所以碰撞在几乎所有目标都是零星的，通常为2k到10k的可发现的分支点

| Branch cnt  |  Colliding tuples | Example targets|
| ---- | ---- | ---- |
|        1,000 | 0.75%            | giflib, lzo|||||
|        2,000 | 1.5%             | zlib, tar, xz||||
|        5,000 | 3.5%             | libpng, libwebp|||
|       10,000 | 7%               | libxml||
|      20,000 | 14%              | sqlite|
|     50,000 | 30%              | -|

同时映射的尺寸足够小到允许以微秒级来分析在接受端并且可以轻松的放入L2 缓存。

这种形式的覆盖范围在程序执行路径中提供了相当多的了解相比与简单的块覆盖，尤其是烦琐的区分以下执行追踪

```
  A -> B -> C -> D -> E (tuples: AB, BC, CD, DE)
  A -> B -> D -> C -> E (tuples: AB, BD, DC, CE)
```

这有助于发现基础代码中的细微故障条件，因为安全漏洞通常与意外事件相关联或错误的状态转换，而不仅仅是到达一个新的基本块。

在之前展示的伪代码的最后一行移位操作时为了保护元组的方向性（不这样做那么A^B 等同于B^A）,和保留循环的标志（不然A^A 和B^B 相同）。

intel处理器缺失简单的饱和算术操作符意味着命中计数器可以回零，因为这是不太可能并且是本地化事件，所有本视为可接受的性能折衷。



# 2 侦察新的行为

fuzzer 保留了之前执行的全局元组映射；这个数据可以很快地和独立的踪迹比较和仅仅用一组dword或qworld指令和一次循环来更更新。

当一个变异的输入产生一次有新元组的执行路径时，对应的输入文件被保留并且稍后给其他处理路由（see section 3 ） . 在执行追踪中不会触发新的本地比例状态过滤的输入将被丢弃。即使他们的总体控制流顺序是唯一的。

这种方法可以进行非常细粒度和长期探索的程序状态，而对任何复杂的执行追踪不必执行任何高强度计算的和脆弱的比较，同时避免路径爆炸的祸害。

为了阐述算法的属性，认为下下面第二条路径会被当做是新的路径，因为存在新的元组。

```
 #1: A -> B -> C -> D -> E
 #2: A -> B -> C -> A -> E

```

同时，当#2被处理了，以下的模式不会被当作是不同的，尽管有明显不同的执行路径。

```
 #3: A -> B -> C -> A -> B -> C -> A -> B -> C -> D -> E
```

除了检测新的元组，fuzz也考虑大概的元组命中率，这些被分为多个桶（组）：

```
 1, 2, 3, 4-7, 8-15, 16-31, 32-127, 128+
```

在某些程度上，桶的数目Shiite是人工实现的，它允许将指令生成的8位计数器映射到fuzzer可执行文件所依赖的8位位图，以跟踪每个元组的已看到的执行计数。

在单个桶范围的变化被忽略； 从一个桶到另一个桶的过滤在程序流中被标记为一次有意思的变化，并且被路由到下面部分概述的进化过程。

命中计数行为提供了一种区分潜在有趣的控制流改变，比如当一个程序块被执行两次当平时只被命中一次，同时

它对经验上不太显著的变化不敏感，比如循环次数从47次到48次。 计数器也提供一些偶然性免疫力来抵抗密集追踪映射的元组爆炸。

执行在很大程度上通过内存和时间来控制，默认情况下，超时被设置为初始校准执行速度的5倍，大约20ms 。过大的超时时间是不值得的，提高百分之一的覆盖率却变慢一百倍，我们务实的拒绝他们并且希望fuuz 可以找到更小花费来到达同样的代码，经验上的测试强烈建议更多的时间是不值得的。

# 3 进化输入队列

相比于更贪心的的遗传算法，这个方法允许工具去逐步探索基础数据格式的不相交和可能不兼容的变异特征，如下图所示，

通过此过程生成的合成语料库实质上是“嗯，这做了新的事情”的 输入文件紧凑集合。并且可以用来为其他任何测试过程提供种子（例如，手动对资源密集型桌面应用进行压力测试）。

使用这种方法，队列中绝大多数的目标发展到其他地方使用大概1k到10k的记录； 大约10%~30%的记录归因与新元组的发现并且被保留的是与命中数的改变有关。

#  4选择语料

上面概述的渐进状态的探索方式意味着过程中稍后一些测试用例的合成可能具有边缘覆盖范围这是由ancestors 提供覆盖范围的严格父集。

为了优化测试效果，AFl 定期的使用一种快速算法来重新变异队列，算法选择了测试用例的更小子集却仍然覆盖至今为止所有的元组，和让他们尤其有利的特征。

这个算法分配给每一个队列记录分配一个与执行延迟和文件大小成比例的分数 ； 然后选择每个租得分最低的候选者。

元组之后就被一个简单的工作流来顺序执行。

1. 在暂时的工作流中发现不存在的元组。
2. 定位该元组的winning（???）队列记录。
3. 在工作集中将记录追踪的当前所有元祖注册。
4. 回到第一步，如果存在任何丢失的元祖

"favored entry"产生的预料通常比刚开始的数据集小5到10倍。Non-favored 记录没有丢弃但是他们在队列中遇到的时候通过各种概率被跳过。
	如果队列中存在新的但尚未fuzz的favorites，则将跳过99%的Non-favored 记录来获取favared 。
	如果没有新的favorites
		如果当前的non-favored记录之前被fuzz过，95%的时间会被跳过（理解95%得non-favored记录被跳过）
		如果还没有完成一轮fuzz，被跳过的概率降到75%.
基于经验测试，这提供了合理的平衡在队列循环速度和测试用例的多样性之间。

可以使用afl-cmin在输入或输出语料库执行稍微更复杂但是更慢的剔除，这个工具可以永久的丢弃冗余的记录和产生更小的语料适合与afl-fuzz或其他外部工具一起使用。

# 5 整理文件输入

文件大小对于fuzz的性能有着巨大的影响，不仅因为运行慢并且降低了降低了触发重要格式控制结构的可能性（不懂没关系后面有），再pref_tips.txt 有更多细节。

除了用户可能会提供低质量的起始语料库的可能性外，某些类型的突变也可能具有迭代增加生成文件大小的作用，因此应对这种趋势很重要。

幸运的是，检测反馈提供了一种自动修剪输入文件的简单方法，同时确保对文件所做的更改不会影响执行路径







1. 尝试将具有较大步长的大数据块归零。 从经验上讲，这可以通过减少以后的细粒度工作来减少执行人员的数量。
2. 使用二进制搜索样式，以减小的块大小和步长执行块删除操作。
3. 通过计算唯一字符并尝试用零值批量替换每个字符来执行字母归一化。
4. 最后的结果是，对非零字节执行逐字节标准化。
